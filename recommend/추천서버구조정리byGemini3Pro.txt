이 프로젝트는 **FastAPI**를 기반으로 한 **이미지 필터 추천 및 검색 마이크로서비스**입니다. 
**CLIP 모델**을 사용해 이미지를 벡터화하고, **ChromaDB**로 벡터 검색을 수행하며, **Redis**로 결과를 캐싱하는 구조를 가지고 있습니다.

---

### 1. 전체 아키텍처 개요

이 서비스는 메인 백엔드(Spring Boot 추정)로부터 데이터를 받아 벡터화하여 저장하고, 
클라이언트의 요청에 따라 추천/검색 결과를 반환하는 **검색/추천 엔진** 역할을 합니다.


#### **데이터 흐름 (Data Flow)**

1.  **인덱싱 (Indexing):**
    * 외부(Spring Boot)에서 `/admin/index` 호출.
    * S3에서 이미지 다운로드 -> CLIP 모델로 벡터 변환(Embedding) -> 메타데이터(스티커 정보 등)와 함께 ChromaDB에 저장 (`upsert`).
2.  **홈 추천 (Recommendation):**
    * 사용자가 좋아한 `filter_ids` 수신 -> Redis 캐시 확인.
    * 캐시 미스 시: ChromaDB에서 입력 필터들의 벡터 조회 -> **평균 벡터(Average Vector)** 계산 -> 유사 벡터 검색(KNN) -> **Re-Ranking(스티커 유사도 반영)** -> Redis 캐싱 -> 결과 반환.
3.  **텍스트 검색 (Search):**
    * 텍스트 쿼리(`q`) 수신 -> CLIP 텍스트 인코더로 벡터 변환 -> ChromaDB 검색 -> 결과 반환.

---

### 2. 모듈별 상세 분석

#### **A. `crud.py` (핵심 비즈니스 로직)**
* **비동기/동기 처리의 조화:** `run_in_threadpool`을 적절히 사용했습니다. CLIP 모델의 `encode()`나 S3 I/O, ChromaDB의 동기 함수들이 FastAPI의 비동기 이벤트 루프(Event Loop)를 차단하지 않도록 별도 스레드에서 실행시킨 점은 **매우 훌륭한 패턴**입니다.
* **S3 처리 전략:** `_fetch_and_vectorize_sync` 함수에서 `requests`로 공개 URL 접근을 먼저 시도하고, 실패 시 `boto3`로 Private S3에 접근하는 **Fallback 로직**이 구현되어 있어 안정적입니다.
* **Numpy 변환 수정:** 주석(`✅ V7 Fix`)에 언급된 대로 PIL 이미지를 `np.asarray`로 변환하여 모델에 주입하는 부분은 데이터 타입 오류를 방지하는 올바른 수정입니다.

#### **B. `services/ranking.py` (랭킹 알고리즘)**
* **하이브리드 추천 로직:** 단순히 벡터 거리(이미지 톤/분위기 유사도)만 보는 것이 아니라, 메타데이터(스티커 배치, 종류, 개수)를 섞어서 **최종 점수**를 산출합니다.
    * **공식:** `최종 점수 = (이미지 유사도 * 0.7) + (스티커 유사도 * 0.3)`
    * **장점:** 이미지가 비슷하더라도 스티커 사용 패턴이 너무 다르면 추천 순위를 낮추는 등 정교한 튜닝이 가능합니다.
* **자카드 유사도(Jaccard Similarity):** 스티커 종류나 배치 타입 같은 집합(Set) 데이터의 유사도를 판단하기 위해 적합한 알고리즘을 사용했습니다.

#### **C. `core/dependencies.py` & `config.py`**
* **Lifespan 관리:** FastAPI의 최신 `lifespan` 컨텍스트 매니저를 사용하여 앱 시작/종료 시 DB 연결과 모델 로딩을 깔끔하게 관리하고 있습니다.
* **환경 설정 분리:** `pydantic-settings`를 사용하여 환경 변수를 타입 안전하게 관리하고 있습니다. 로컬 테스트와 프로덕션(IAM Role)을 구분하는 로직도 `crud.py`에 잘 반영되어 있습니다.

### 3. 코드 품질 및 개선 제안 (Code Review)

전반적으로 코드가 깔끔하고 모듈화가 잘 되어 있습니다. 하지만 상용 배포(Production)를 고려했을 때 몇 가지 개선할 포인트가 보입니다.

#### **✅ 잘된 점 (Pros)**
1.  **Blocking 방지:** CPU 집약적인 작업(모델 추론)을 `run_in_threadpool`로 격리하여 서버의 응답성을 유지함.
2.  **캐싱 전략:** Redis를 도입하여 고비용 연산(벡터 검색)의 빈도를 줄임.
3.  **명확한 책임 분리:** `crud`, `models`, `services`로 역할이 명확히 나뉘어 있음.

#### **⚠️ 개선이 필요한 점 & 리스크 (Cons & Risks)**

**1. Re-Rank 로직의 단순화 (Critical)**
* **현재 로직:** `re_rank_candidates` 함수에서 입력 필터가 여러 개일 때, **첫 번째 필터(`input_metadatas[0]`)의 스티커 정보만 기준**으로 삼고 있습니다.
    * *문제점:* 사용자가 필터 A(스티커 많음)와 필터 B(스티커 없음)를 모두 좋아했는데, A가 리스트의 첫 번째라면 스티커가 많은 필터 위주로만 Re-rank 됩니다.
    * *해결:* 입력된 모든 필터의 스티커 속성을 취합(Aggregate)하거나, 각 후보 필터에 대해 입력 필터들과의 최대 유사도를 구하는 방식으로 개선해야 합니다.

**2. CLIP 모델 로딩 속도 및 메모리**
* 현재 `SentenceTransformer`를 CPU 모드로 로딩하고 있습니다. 트래픽이 몰릴 경우 CPU 부하가 심해질 수 있습니다.
* *해결:* 추후 GPU 인스턴스로 마이그레이션하거나, 모델을 ONNX 런타임으로 변환하여 경량화/가속화하는 것을 고려해보세요.

**3. 페이징(Pagination) 로직의 효율성**
* **현재:** `get_text_search_results`에서 `n_results=settings.PAGE_SIZE * (page + 1)`로 쿼리한 뒤 Python 리스트 슬라이싱으로 자릅니다.
* *문제:* 페이지가 뒤로 갈수록(예: 10페이지) ChromaDB에서 가져오는 데이터 양이 선형적으로 늘어납니다 (200개 -> 400개...).
* *해결:* ChromaDB는 `offset` 파라미터를 지원하지 않는 경우가 많아 어쩔 수 없는 선택일 수 있지만, `PRIMARY_QUERY_COUNT`를 고정하고 그 안에서 페이징을 처리하는 전략이 성능상 더 유리할 수 있습니다.

**4. 예외 처리의 구체화**
* `crud.py`의 `except Exception as e:` 구문이 다소 포괄적입니다. S3 연결 오류, 모델 추론 오류, DB 연결 오류를 구분하여 로그를 남기면 디버깅이 훨씬 수월해집니다.

### 4. 다음 단계 제안 (Action Plan)

현재 코드는 MVP(Minimum Viable Product)로서 기능적으로 훌륭합니다. 프로젝트의 완성도를 높이기 위해 다음 작업을 추천합니다.

1.  **Re-Rank 로직 고도화:**
    * 입력된 필터 리스트(`filter_ids`) 전체의 스티커 특성을 반영하도록 `services/ranking.py` 수정.
    * (예: 입력 필터들의 스티커 타입 합집합을 기준(Base)으로 사용)
2.  **Docker화 및 배포 준비:**
    * `requirements.txt`에 `torch`, `sentence-transformers` 등 무거운 라이브러리가 포함되므로, Docker 빌드 시 캐싱을 효율적으로 사용하는 `Dockerfile` 작성이 필요합니다.
3.  **테스트 코드 작성:**
    * 특히 `calculate_sticker_similarity` 같은 로직은 단위 테스트(Unit Test)를 통해 의도한 대로 점수가 계산되는지 검증해야 합니다.
